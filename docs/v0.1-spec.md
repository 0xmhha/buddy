# Buddy v0.1 — Reliability Wedge Spec

> 목적: Claude Code 운영의 *조용한 실패(silent failure)*를 잡고, 세션 상태를
> schema로 검증하며 보존한다. 다른 모든 기능(orchestration, dashboard)의 신뢰
> 기반이 된다.
> 작성일: 2026-04-23 / 상태: DRAFT (사용자 결정 3건 대기 — §6)

---

## 0. 한 문단 요약

Buddy v0.1은 **3가지 일**만 한다:

1. **Hook health 감시** — Claude Code의 `~/.claude/settings.json`에 등록된 hook
   실행을 가로채 latency·exit code·timeout을 기록하고, 임계값을 넘으면 알린다.
2. **State store + schema** — buddy 자신의 모든 상태(hook 통계·세션 인덱스·user
   preference)를 SQLite + Zod 스키마로 저장하여 손상·중복을 막는다.
3. **WAL replay** — buddy가 죽어도 기록되지 않은 이벤트를 재처리하여 통계 손실
   을 방지한다.

이 3가지가 갖춰지면 v0.2(dashboard)와 v0.3(orchestration)이 *신뢰할 수 있는
상태 위에서* 작동할 수 있다.

## 3. 기술 스택 결정 (2026-04-24 PIVOT: TS → Go)

| 항목 | 선택 | 근거 |
|------|------|------|
| Language | **Go 1.22+** | hook wrapper self-overhead 핵심 (Node 50-100ms vs Go 5-10ms) |
| State | **SQLite (`modernc.org/sqlite`)** | pure Go, cgo 불필요, 사내 빌드 환경 안전 |
| Schema | **struct + Validate() 메서드** | 의존성 0, 경계 검증만 |
| WAL | SQLite WAL mode + 자체 outbox 테이블 | cli-wrapper 패턴 차용 |
| CLI | **`spf13/cobra`** | Go 표준, 풍부한 subcommand 지원 |
| Test | **`stretchr/testify`** + 표준 testing | cli-wrapper와 동일 |
| Process supervision | **`0xmhha/cli-wrapper` (M3+)** | daemon lifecycle/restart/WAL을 import로 위임 |
| Dist | static binary (cross-compile matrix) | npm/Node 의존성 0 |

**Pivot 근거 (Opinion, High):**
1. Hook wrapper는 매 hook 호출마다 spawn → startup 비용이 self-overhead의 본질
2. cli-wrapper는 분석 보고서 §3 "재사용 가치 Top 5" — Go이면 native import
3. 폐기 비용은 *지금이 가장 낮음* (M2까지만 작성 → 920 LOC)
4. *설계 사고는 100% 보존* (schema, invariants, outbox 패턴 모두 1:1 이식)

**보존 자산:** `archive/ts-poc/` 에 TS PoC 전체 보존 (참조용).

## 4. 아키텍처 (M3 반영)

```
┌──────────────────────────────────────────────────────────┐
│                     Claude Code                          │
│  hook event ──▶  bash command from settings.json         │
└──────────────────┬───────────────────────────────────────┘
                   │ (1) buddy hook-wrap로 감싸짐
                   ▼
┌──────────────────────────────────────────────────────────┐
│  buddy hook-wrap (Go binary, ~5-10ms cold start)         │
│  - stdin parse + child spawn (streaming passthrough)     │
│  - duration/exit code 측정                               │
│  - hook_outbox에 sync INSERT (원본 hook 종료 후)         │
└──────────────────┬───────────────────────────────────────┘
                   │ (2) outbox row
                   ▼
┌──────────────────────────────────────────────────────────┐
│  buddy daemon (poll 1s default)                          │
│  - aggregator.ProcessBatch() per tick                    │
│    · hook_outbox → hook_events (정규화 INSERT)           │
│    · hook_events → hook_stats (5/60/1440 min 윈도우)     │
│    · outbox row consumed_at mark                         │
│  - 단일 goroutine, SIGTERM 시 graceful exit              │
│  - PID file (flock) → 중복 실행 방지                     │
└────────────────┬─────────────────────────────────────────┘
                 │ supervised by (선택)
                 ▼
┌──────────────────────────────────────────────────────────┐
│  cli-wrapper (`pkg/cliwrap`) — hybrid 통합               │
│  - buddy 단독 실행 가능 (`buddy daemon start`)           │
│  - 또는 cliwrap.yaml로 supervise (자동 재시작/리소스 budget) │
│  - M4의 `buddy install --with-cliwrap`이 yaml 자동 생성  │
└──────────────────────────────────────────────────────────┘
                 ▲
                 │ (3) buddy CLI (read-only)
                 │
                 user: doctor / stats / events
```

**핵심 invariant:**
- (1) hook-wrap은 원본 hook 종료 후 outbox에 *sync* INSERT → buddy 자신이
  hook 지연 원인이 되어도 데이터 손실 0. outbox 실패는 hook을 깨지 않는다.
- (2) daemon은 옵셔널. 없어도 outbox는 쌓임 → daemon 시작 시 1회 즉시 tick으로 catch-up.
- (3) CLI 조회는 SQLite read-only로 직접 접근 (daemon 의존 X).
- (4) **hybrid supervision**: buddy 단독으로도 작동. cli-wrapper는 *가산점*이지 *전제*가 아님.

## 5. 데이터 모델 (DRAFT — 사용자 결정 §6.1 필요)

### 5.1 핵심 테이블

```sql
-- WAL outbox: hook wrapper가 sync로 쓰고, daemon이 비동기로 비움
CREATE TABLE hook_outbox (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  ts          INTEGER NOT NULL,           -- unix ms
  payload     TEXT NOT NULL,              -- JSON (Zod validated downstream)
  consumed_at INTEGER                     -- NULL = pending
);
CREATE INDEX idx_outbox_pending ON hook_outbox(id) WHERE consumed_at IS NULL;

-- 정규화된 이벤트
CREATE TABLE hook_events (
  id          INTEGER PRIMARY KEY,
  ts          INTEGER NOT NULL,
  event       TEXT NOT NULL,              -- PreToolUse / PostToolUse / Stop / ...
  hook_name   TEXT NOT NULL,              -- 사용자가 settings.json에 정의한 키
  duration_ms INTEGER NOT NULL,
  exit_code   INTEGER NOT NULL,
  session_id  TEXT,                       -- Claude Code session id (있으면)
  pid         INTEGER,
  cwd         TEXT,
  -- 추가 필드는 §6.1 결정 후 확정
  meta        TEXT                        -- JSON, schemaless 확장 슬롯
);
CREATE INDEX idx_events_hook_ts ON hook_events(hook_name, ts);

-- 집계 (rolling)
CREATE TABLE hook_stats (
  hook_name   TEXT NOT NULL,
  window_min  INTEGER NOT NULL,           -- e.g., 5/60/1440
  ts_bucket   INTEGER NOT NULL,
  count       INTEGER NOT NULL,
  failures    INTEGER NOT NULL,
  p50_ms      INTEGER,
  p95_ms      INTEGER,
  p99_ms      INTEGER,
  PRIMARY KEY (hook_name, window_min, ts_bucket)
);
```

### 5.2 Zod 스키마 (TS)

```typescript
import { z } from 'zod'

// hook wrapper가 outbox에 쓰는 payload
export const HookEventPayload = z.object({
  ts: z.number().int().positive(),
  event: z.enum([
    'SessionStart', 'PreToolUse', 'PostToolUse',
    'Stop', 'PreCompact', 'UserPromptSubmit'
  ]),
  hookName: z.string().min(1).max(100),
  durationMs: z.number().int().min(0),
  exitCode: z.number().int(),
  sessionId: z.string().uuid().optional(),
  pid: z.number().int().positive().optional(),
  cwd: z.string().optional(),
  meta: z.record(z.unknown()).optional(),  // schemaless 확장
})
export type HookEventPayload = z.infer<typeof HookEventPayload>
```

### 6.3 Buddy의 페르소나 / 말투 ✅ DECIDED (2026-04-23)

**선택: B) 친구 톤 (한국어 기본)**

```typescript
// src/cli/persona.ts
export const persona = {
  // 정상 상태 = 침묵 (success quiet)
  // critical = 절제된 친구 톤 (호들갑 X)
  // info     = 친구 톤
  // debug    = 무미건조 (구조적 로그)
}
```

**예시 메시지:**

| 상황 | 메시지 |
|------|--------|
| slow hook 감지 | `어, 'pre-commit' hook이 좀 느려졌어. p95가 8.2초 (기준 5초).` |
| 실패율 임계 초과 | `'lint' hook 실패율이 35% 야. 최근 20번 중 7번 실패.` |
| outbox backlog | `outbox에 1,247개 쌓였어. daemon 한 번 봐줘 (buddy daemon status).` |
| timeout | `'build' hook이 30초 넘게 멈춰있어. PID 12345.` |
| daemon 정상 시작 | (침묵) |
| install 성공 | `등록 완료. 이제 옆에서 보고 있을게.` |

**원칙:**
1. **침묵이 default** — 정상 상태는 출력 없음
2. **critical도 절제** — "🚨🚨" 같은 호들갑 금지
3. **이모지는 안 씀** — 친구지만 차분한 친구
4. **debug 로그는 구조적** — `~/.buddy/logs/`는 JSON, 친구 톤 X
5. **영어 fallback** — locale이 ko가 아니면 같은 의미 영어. v0.1은 ko/en만

## 7. CLI 표면 (v0.1)

### 7.1 hook-wrap ✅ M2 IMPLEMENTED

```
buddy hook-wrap <hook-name> [original-command...]
  --event <name>          기본 event (stdin 비었을 때 fallback, default: PreToolUse)
  --db <path>             buddy DB 경로 (default: ~/.buddy/buddy.db)
  --record-tool-args      tool_input을 outbox에 기록 (default: off, privacy)
  --tag key=value...      customTags 추가 (반복 가능)
```

**Invariants (v0.1 보장):**
1. **stdout 침묵**: wrapper는 자기 출력을 stdout에 절대 쓰지 않는다. 모든
   buddy 메시지는 stderr 또는 로그.
2. **streaming passthrough**: 원본 hook의 stdout/stderr는 buffering 없이
   부모 stdio에 직접 pipe.
3. **stdin 재전달**: stdin은 buddy가 한 번 buffering(=parse용)한 뒤 child에
   다시 흘려준다.
4. **exit code 통과**: 원본 hook의 exit code를 그대로 전달. signal 종료는
   `128 + signal_no`로 변환.
5. **outbox 실패는 hook을 깨지 않는다**: outbox write 실패 시 stderr에 한
   줄만 남기고 hook의 exit code 통과.
6. **child가 없으면 monitoring-only**: `original-command`가 비면 wrapper는
   stdin parse + outbox 기록만 하고 exit 0.
7. **알 수 없는 hook input은 흡수**: malformed JSON이나 미래 필드는 무시
   하고 진행 (wrapper 자체는 절대 깨지지 않는다).

### 7.2 daemon (M3 IMPLEMENTED)

```
buddy daemon run           # foreground (cli-wrapper supervise용)
  --db <path>              # buddy DB 경로
  --pid <path>             # PID 파일 경로
  --poll <duration>        # outbox poll 간격 (기본 1s)
  --batch <int>            # tick당 처리 상한 (기본 500)

buddy daemon start         # detach fork (단독 실행)
buddy daemon stop          # SIGTERM
buddy daemon status        # PID + liveness 보고
```

### 7.3 install / doctor / stats / events (M4 IMPLEMENTED)

```
buddy install                  # ~/.claude/settings.json hook을 buddy hook-wrap으로 감싼다
  --claude-dir <path>          # default: ~/.claude
  --buddy-dir <path>           # default: ~/.buddy
  --buddy-binary <path>        # default: 현재 실행 파일
  --db <path>                  # cliwrap.yaml 안의 daemon --db 값 (이후 명령에 자동 적용 X)
  --with-cliwrap               # cliwrap.yaml도 함께 생성 (~/.buddy/cliwrap.yaml)

buddy uninstall                # 백업 우선 복원, 백업 없으면 JSON walk로 unwrap
  --claude-dir / --buddy-dir / --buddy-binary

buddy doctor                   # 1회 health snapshot (read-only). exit 0 healthy, 1 issue
  --db <path> / --pid <path>

buddy stats                    # hook_stats 조회 + 친구 톤 표
  --window <5m|1h|24h>         # default 1h
  --by-tool                    # tool 단위 row (기본은 hook 단위 합산)
  --hook <name>                # case-insensitive 필터
  --db <path>

buddy events                   # hook_events tail (debug 한 줄 포맷)
  --limit <int>                # default 20 (>= 1)
  --hook <name>                # case-insensitive 필터
  --follow / -f                # 1s poll, ctx 취소 시 종료
  --db <path>
```

### 7.4 config / purge (M5 IMPLEMENTED)

```
buddy config show [--json] [--config <path>]                # 효과적 config 출력 (default + override)
buddy config get <field> [--config <path>]                  # 단일 필드 값 (shell 치환 친화)
buddy config set <field> <value> [--config <path>]          # 변경 + 검증 + 원자적 저장 (silent on success)
buddy config unset <field> [--config <path>]                # override 해제 (default로 되돌림)

buddy purge --before <date> [--apply] [--db <path>]
  # <date>: 30d / 2026-04-01 / 2026-04-01T00:00:00Z
  # default: dry-run (count). --apply 시 실제 삭제
  # hook_outbox 는 절대 건드리지 않는다 (§4 invariant 1)
```

config schema (`~/.buddy/config.json`): hookTimeoutMs / hookSlowMs /
hookFailRatePct / outboxBacklog / notifyChannel / pollInterval / batchSize /
personaLocale. 결정 우선순위: 명시적 CLI flag > config 파일 > spec-locked
default. `doctor` / `daemon run|start` 도 `--config <path>` 수용.
